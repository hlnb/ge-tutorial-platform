# Getting a Website Online: Hosting, Deployment and Continuous Improvement

A website is not truly finished when it works on your computer.

That is only the “it works in my room” stage.

Deployment is the process of putting the website somewhere other people can access it through a browser. It is the point where your files leave the cosy little nest of your laptop and meet the wider internet.

This is also where hidden problems often appear: broken image paths, missing files, incorrect folder structure, case-sensitive filenames, and pages that only worked because they were sitting on your own machine.

In this lesson, you will learn how deployment works, how different hosting options compare, and how to create a simple workflow for improving your website after it goes live.

## Learning objectives

By the end of this tutorial, you should be able to:

- explain what deployment means
- describe the difference between a local website and a live website
- identify the basic parts of web hosting
- understand traditional hosting, manual static hosting, and Git-based deployment
- deploy a simple HTML, CSS and JavaScript website using an appropriate method
- understand how Vite changes the deployment process
- use a pre-deployment checklist before publishing
- create a simple continuous improvement workflow
- recognise the difference between student practice hosting and production-ready hosting

## What deployment means

Deployment means publishing a website so it can be viewed online.

For a basic static website, the browser needs access to the site files:

- HTML files
- CSS files
- JavaScript files
- images
- fonts
- other assets

Your local project folder is like the kitchen.

Deployment is putting the finished meal on the table.

The browser cannot access your laptop directly. The website needs to be placed on a server or hosting platform that is connected to the internet.

## Key terms

| Term | Meaning |
|---|---|
| Local site | The version of the website stored on your own computer. |
| Live site | The version of the website available online. |
| Web host | A service that stores and serves website files. |
| Domain name | The human-friendly address for a website. |
| Server | A computer that sends website files to visitors. |
| Deployment | The act of publishing or updating the live site. |
| Static site | A website made from files that do not need server-side processing. |
| Build | The process of preparing a development project for production. |
| Output folder | The folder generated by a build process, such as `dist` in many Vite projects. |

## Local files versus live websites

When you are learning HTML, CSS and JavaScript, you may open `index.html` directly from your computer and see the page in your browser.

That is useful for early testing. It is not the same as deploying the site.

Local viewing can hide problems such as:

- file paths that only work on your machine
- filenames that fail on case-sensitive servers
- missing files hidden by browser cache
- forms that do not behave as expected
- routing issues
- image paths that point to the wrong location
- mobile layout problems
- JavaScript errors that only appear after deployment

The site working on your laptop is encouraging. It is not yet evidence.

Deployment is where the browser gets brutally honest.

## Checkpoint: Is this ready to leave your laptop?

Before you deploy, check:

- [ ] Does the homepage open correctly?
- [ ] Do all navigation links work?
- [ ] Are images stored inside the project folder?
- [ ] Are filenames simple, consistent and lowercase?
- [ ] Are CSS files linked correctly?
- [ ] Are JavaScript files linked correctly?
- [ ] Does the site work on a phone-sized screen?
- [ ] Are there no obvious console errors?
- [ ] Is the main page named `index.html`?
- [ ] Have you removed placeholder content?

## The three main hosting paths

There are many hosting services, but beginner deployment usually falls into three broad paths.

### Option A: Traditional web hosting

Traditional web hosting usually gives you a dashboard or control panel, a file manager, and sometimes FTP access.

You upload your HTML, CSS, JavaScript and image files into a public website folder.

Common public folder names include:

- `public_html`
- `htdocs`
- `www`

This is useful for students who want to understand how basic hosting works without GitHub.

#### Traditional hosting is best for

- learning how hosting folders work
- simple static sites
- small manual projects
- understanding what many small business hosting accounts look like
- seeing where website files actually live on a host

#### Traditional hosting limitations

- manual updates can become messy
- it is easy to overwrite files
- there is no version history unless you keep backups
- it is less suitable for modern development workflows
- free hosting may have limits, slower performance or limited support

Manual hosting teaches you where the files go.

### Option B: Manual static hosting

Some modern static hosting services let you upload a finished folder manually.

Examples include:

- Netlify Drop
- Vercel Drop
- Cloudflare Pages Direct Upload

This is useful when you want quick publishing without learning GitHub yet.

#### Manual static hosting is best for

- prototypes
- class exercises
- simple HTML, CSS and JavaScript demos
- quick portfolio experiments
- testing whether a project is ready for the web

#### Manual static hosting limitations

- updates still require manual uploads
- you must remember exactly which folder you uploaded
- multiple versions can become confusing
- it is not as repeatable as Git-based deployment

### Option C: Git-based deployment

Git-based deployment connects a GitHub repository to a hosting service.

When you push changes to GitHub, the hosting service automatically rebuilds and redeploys the website.

Examples include:

- GitHub Pages
- Netlify connected to GitHub
- Vercel connected to GitHub
- Cloudflare Pages connected to GitHub

#### Git-based deployment is best for

- ongoing projects
- portfolio sites
- Vite projects
- team workflows
- version history
- continuous improvement
- repeatable deployment

#### Git-based deployment limitations

- you need basic Git and GitHub confidence
- setup can feel abstract at first
- build settings matter
- error logs can look intimidating until you learn how to read them

Git-based deployment teaches you how modern websites keep improving without becoming digital spaghetti.

## Plain HTML, CSS and JavaScript deployment

For a plain static website, your project might look like this:

~~~text
my-website/
├── index.html
├── about.html
├── styles.css
├── script.js
└── images/
    └── hero.jpg
~~~

To deploy this kind of site manually, you usually upload the files as they are.

The most important rule is this:

`index.html` must be in the correct public folder.

If you upload the whole `my-website` folder into the hosting folder, your homepage may end up one level too deep.

For example, this may fail at the main address:

~~~text
htdocs/
└── my-website/
    └── index.html
~~~

This is usually better:

~~~text
htdocs/
├── index.html
├── about.html
├── styles.css
├── script.js
└── images/
    └── hero.jpg
~~~

The server looks for `index.html` first. Do not make it go on a treasure hunt.

## Vite deployment

A Vite project is different from a plain HTML folder.

During development, you usually run:

~~~bash
npm run dev
~~~

That starts a local development server.

Before deployment, you usually run:

~~~bash
npm run build
~~~

This creates a production-ready version of the site. In many Vite projects, the output folder is:

~~~text
dist/
~~~

For manual deployment, you usually upload the contents of the `dist` folder or the `dist` folder itself, depending on the hosting service instructions.

For Git-based deployment with Netlify or Vercel, you usually set:

| Setting | Value |
|---|---|
| Build command | `npm run build` |
| Output / publish directory | `dist` |

Do not upload the whole development project folder to a manual host and expect it to behave like the finished site.

Your source files are the workshop.

Your `dist` folder is the packed and polished version ready for visitors.

## Guided practice: Deploy a tiny test site

Before deploying a larger project, create and deploy a tiny test website.

This keeps the first deployment low-pressure. If something breaks, there are only three files to inspect.

Create this folder structure:

~~~text
deployment-test/
├── index.html
├── styles.css
└── script.js
~~~

Create `index.html`:

~~~html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>Deployment Test</title>
  <link rel="stylesheet" href="styles.css">
</head>
<body>
  <main>
    <h1>My first deployed test site</h1>
    <p>If you can read this online, deployment worked.</p>
    <button id="messageButton">Test JavaScript</button>
    <p id="message"></p>
  </main>

  <script src="script.js"></script>
</body>
</html>
~~~

Create `styles.css`:

~~~css
body {
  font-family: system-ui, sans-serif;
  margin: 0;
  padding: 2rem;
  background: #f7f3ed;
  color: #222;
}

main {
  max-width: 42rem;
  margin: 0 auto;
}

button {
  padding: 0.75rem 1rem;
  border: 0;
  border-radius: 0.5rem;
  cursor: pointer;
}
~~~

Create `script.js`:

~~~js
const button = document.querySelector('#messageButton');
const message = document.querySelector('#message');

button.addEventListener('click', () => {
  message.textContent = 'JavaScript is working on the live site.';
});
~~~

Deploy this test site using one method:

- traditional hosting
- Netlify Drop
- Vercel Drop
- GitHub Pages
- Netlify with GitHub
- Vercel with GitHub

Then complete the deployment checklist.

## Continuous improvement workflow

Deployment is not the end of the work.

It is the beginning of maintaining the site.

A simple improvement workflow prevents you from randomly changing files and hoping for the best.

### Manual hosting workflow

Use this flow:

~~~text
Notice issue → Write it down → Fix local file → Test locally → Upload changed file → Test live site → Record the change
~~~

Each step matters.

| Step | Why it matters |
|---|---|
| Notice issue | You identify what needs to change. |
| Write it down | You avoid relying on memory. Memory is not a project management system. |
| Fix local file | You update the source version first. |
| Test locally | You catch obvious mistakes before uploading. |
| Upload changed file | You update the live site. |
| Test live site | You confirm the real website works. |
| Record the change | You create a history of what happened. |

### Git-based workflow

Use this flow:

~~~text
Notice issue → Create issue or task → Fix locally → Test locally → Commit → Push → Deployment runs → Test live site
~~~

GitHub-connected deployment creates a cleaner trail of changes.

You can see what changed, when it changed, and often why it changed.

That is helpful when something breaks and the website starts behaving like it has joined a witness protection program.

## Student improvement log

Use a simple table like this:

| Date | Issue or improvement | Change made | Tested locally? | Tested live? | Notes |
|---|---|---|---|---|---|
| YYYY-MM-DD | Navigation link broken | Fixed file path | Yes | Yes | Link now works |
| YYYY-MM-DD | Image too large | Compressed image | Yes | Yes | Page loads faster |

## Common deployment problems

### Problem: My homepage does not load

Likely causes:

- `index.html` is missing
- `index.html` is inside the wrong folder
- the wrong folder was uploaded
- hosting account has not finished setting up

Fix:

- check the public folder
- move `index.html` to the correct location
- refresh the live URL
- wait a few minutes if the host is still provisioning the site

### Problem: CSS is missing

Likely causes:

- CSS file path is wrong
- CSS file was not uploaded
- filename uses different capitalisation
- CSS folder structure changed

Fix:

- check the `<link>` path
- check the uploaded folder
- match file and folder names exactly

### Problem: Images are broken

Likely causes:

- image files were not uploaded
- local file paths were used
- folder name mismatch
- uppercase/lowercase mismatch
- spaces in filenames

Fix:

- use relative paths
- use simple lowercase filenames
- keep images in the project folder
- re-upload missing images

### Problem: The site works locally but not online

Likely causes:

- local file paths
- incorrect folder uploaded
- case-sensitive filename issue
- Vite project source uploaded instead of build output
- build settings incorrect

Fix:

- check paths
- check public folder
- run the build command if using Vite
- upload `dist` if deploying manually
- check build logs if using Git deployment

### Problem: Updates do not appear

Likely causes:

- browser cache
- wrong site updated
- wrong folder uploaded
- deployment still processing
- Git push did not happen

Fix:

- hard refresh
- check the deployment dashboard
- check the live URL
- confirm the correct folder or repository
- test in a private browser window

Deployment means putting a website online.

Free hosting is useful for learning, but it has limits.

GitHub integration helps with repeatable deployment and continuous improvement.
